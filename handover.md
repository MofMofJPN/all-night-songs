## 2. 開発思想

### 2.1 最優先事項

#### ① HSPへの配慮
ゑうたん本人がHSP気質を自認している。リスナーにもHSPの人が多い前提で設計：
- 派手な点滅・ストロボ効果は禁止
- 急激な動きはNG、すべてのアニメーションを穏やかに
- 強い色・コントラストを避ける
- 余白を多く取る
- 「何かを強制される」UIにしない（モーダルの押し付け等）

#### ② スマホファースト
主な利用シーン: **配信中にリスナーがスマホで使う**
- 縦画面前提
- タップしやすいボタンサイズ
- 片手操作可能なレイアウト
- 暗い環境（夜の配信視聴）でも見やすい配色

#### ③ 軽量・即起動
配信中にサッと開いて使う想定。重い読み込みは離脱の原因になる：
- 外部ライブラリ最小限
- 画像も最小限（SVGアイコン主体）
- LocalStorage活用でサーバー通信不要
- 静的サイトとして公開（GitHub Pages）

### 2.2 「やらない」と決めたこと（重要）

#### Claude APIによるAI推薦
**理由**: コストが累積する。リスナーが何度も使うとAnthropicへの課金が無視できなくなる。
**代替**: タグベースのマッチングで十分。3問のアンケート程度なら静的ロジックで対応可能。

#### バックエンド初期構築
**理由**: 開発・運用コストが跳ね上がる。最初は需要が読めない。
**代替**: LocalStorageで開始。需要が見えてからGoogle Forms連携 → Supabase移行を検討。

#### リアルタイム集計・ランキング
**理由**: バックエンドが必要。最初は利用者も少ない。
**代替**: 当面は個人ごとのLocalStorage完結。Phase 2でGoogle Forms+手動週次集計を検討。

#### 全曲フィルタを必須にすること
**理由**: 「3問のアンケート」がメイン体験。フィルタは補助。
**実装**: 全曲リストにはフィルタを置くが、トップ画面のメイン導線はクイズ。

#### リスナー名プリセット機能
**理由**: 議論の途中で「不要」と判断。リクエスト文に名前を自動挿入する機能は冗長。
**実装**: リクエスト文はシンプルなコピペテンプレートのみ。

#### ゑうたん本人おすすめコーナー
**理由**: 本人の協力が前提になる機能。一ファンとしては実装の責任が持てない。
**実装**: 検討中機能リストからも削除。将来コラボの機会があれば検討。

#### 配信ごとの「歌った曲」アーカイブ
**理由**: 配信を毎回追跡してデータ更新する運用コストが高い。一ファン運営の限界を超える。
**実装**: 検討中機能リストからも削除。

### 2.3 検討中の機能（Aboutに記載するもの）
将来要望があれば検討する機能として、Aboutページに以下2項目だけ明示：

- 🌙 クラウド保存（端末をまたいで使えるように）
- 🌙 みんなのお気に入りランキング

「これ欲しい！」の窓口表記は **入れない**（一ファン運営のため、対応コミット感を出さない）。

## 3. デザイン決定の経緯

### 3.1 当初検討した方向性

**最初の候補**: ネオンサイバー × ダークモード × 紫
- 配信文化との親和性
- クール・スタイリッシュ感

**修正のきっかけ**:
ゑうたんのリットリンクと誕生日プロフィールを確認した結果：
- 本人の世界観は「ネオンサイバー」より「**やわらかい夜空**」だった
- 紫はラベンダー寄りの優しいトーン
- HSP気質で派手な刺激を好まない
- 「ALL NIGHT」というキャッチが既に使われていた

### 3.2 最終的なコンセプト

**「夜空の音楽図書館」**

- 深夜にしっとり曲を選ぶ体験
- 派手なサイバー感はNG
- リットリンクと統一感のあるトーン
- 「夜空・星・カラスのシルエット」モチーフ

### 3.3 配色の根拠

リットリンク・誕生日プロフィール画像から逆算：

| 色 | hex | 由来 |
|---|---|---|
| 背景 | #1a1a2e | 夜の街並みの濃紺 |
| カード | #2a2645 | リットリンクのボタン色系統 |
| メインテキスト | #f5efd9 | 誕生日プロフィールの星のクリーム色 |
| プライマリ | #9b85c4 | リットリンクのボタンと同じ薄紫 |
| 差し色 | #e8c873 | 窓明かりの黄（ごく一部のみ使用） |

### 3.4 サイト名・キャッチコピーの決定経緯

- 「ALL NIGHT SONGS」: ゑうたん本人が誕生日プロフィールで使っている「ALL NIGHT」を継承
- 「ゑうたん歌リスト」: 機能をシンプルに説明するサブタイトル
- 「1曲どうぞ」: 短くて多義的。配信文化の「1曲お願いします」のニュアンス＋配信者からの「どうぞ」の優しさ

### 3.5 カラスモチーフの扱い

- ゑうたんファン界隈の象徴「カラス＋✨」をサイトに反映
- ただし**控えめに**（HSP配慮、派手にしない）
- 各画面のヘッダー右上に小さく配置
- 色はラベンダー紫、opacity 0.6〜0.7程度
- サイズは50px前後（議論経て調整済み）

#### 重要：プレースホルダー方式で実装

**現状**: Claude Designで自動生成されたカラスシルエットが意図と違ったため、
画像を **プレースホルダー** に置き換えている。

**理由**:
- 後日 Motobu が PNG画像を用意して差し替える予定
- レイアウトの占有スペースを確保しておくことで、画像投入時に他要素がズレない
- 「ここにカラスが入る」という意図を視覚的に残す

**実装方針**:
- 配置場所・サイズはプレースホルダーで確保（全画面のヘッダー右上）
- CSSの `--crow-image` 等の変数にして、**1箇所の変更で全画面に反映**できる構造に
- Tweaksパネルの「カラスを表示」トグルは維持

**将来 PNG を差し替える時の手順**:
1. PNG画像を用意（推奨スペック：透過PNG、高さ120〜200px、ラベンダー紫または白の単色ベース、横向き構図）
2. Claude Design または Claude Code で画像をアップロード
3. 「現在プレースホルダーになっているカラスの位置すべてに、このPNGを適用」と指示
4. 全画面が一気に更新される

**カラスの隣の ✨ について**:
- ファンマーク「カラス＋✨」を再現するため、PNG差し替え時に小さく1〜2個の✨を追加検討
- 星の背景と区別できるよう、わずかに大きめ・色味を変える

## 4. データ構造の設計判断

### 4.1 タグ軸を3軸に絞った理由

候補だった軸:
- 気分・感情
- テンポ
- シーン
- ジャンル
- 季節
- 曲調（明・暗）

最終的に「**気分 / テンポ / シーン**」の3軸を採用。

**理由**:
- 「3問のアンケート」がメイン体験なので、軸を3つに揃えるのが自然
- ジャンルは「リスナーの気分」とズレやすい（「ロック気分」より「盛り上がりたい気分」のほうが直感的）
- 季節は配信時期に依存しすぎる

### 4.2 タグ重み付け（気分2点、テンポ1点、シーン1点）の理由

完全均等にしなかった理由:
- 「気分」が一番リスナーの聴きたいものに直結する
- テンポやシーンは「あれば嬉しい」程度の補助
- 重み付けによって「気分」軸の合致を優先できる

### 4.3 表示曲数を5曲にした理由

候補: 3曲 / 5曲 / 10曲

選択肢の心理学的な「選びやすい上限」を意識:
- 3曲だとピンとこない時の代替がない
- 10曲だと多すぎて選べない
- 5曲がバランス良い

### 4.4 「聴いた記録から候補除外」をしない判断

最初は「聴いた曲は次の診断結果から外す」設計だった。
**変更理由**:
- 歌ってみた文化は「**同じ曲を何度でも聴きたい**」側面が強い
- 「あの人が歌う春泥棒、また聴きたい」という気持ちを尊重したい

**現在の仕様**:
- 「聴いた記録」は **思い出コレクション**として機能（候補から外さない）
- 重複回避は別の仕組み（**直前の診断結果5曲のみ**次回除外）で実現

### 4.5 タグ付け作業の進め方

156曲をタグ付けする際の役割分担:
1. Claude（チャット）が一次スクリーニング（◎自信あり / ◯まあまあ / △推測）
2. △の38曲を Perplexity / Gemini に検索プロンプトで投げる
3. 検索AIの結果を統合して最終データ確定

この方法で**速度と精度の両立**を実現。今後の曲追加時も同じフローを推奨。

## 5. 段階的なロードマップ

### Phase 1: ローンチ（現在の実装範囲）
- 7画面の静的サイト
- LocalStorageで個人データ管理
- GitHub Pagesで無料公開
- 156曲スタート

### Phase 2: データ収集（リリース後1〜2週間後検討）
**条件**: 利用者が定着し始めたら

**実装内容**:
- Google Forms連携で集計用データ収集
- スプレッドシートに蓄積
- 週次手動集計で「今週のお気に入りTOP5」をJSONで更新
- サイトに表示

**判断基準**:
- リスナーから「集計見たい」の声が来る
- 配信で繰り返し話題に出る

### Phase 3: バックエンド本格化（要望次第）
**条件**: 以下のサインが2つ以上揃ったら

- 定期利用者が15人を超えてきた
- 「データ消えた」「別端末で見たい」の声が複数届く
- 毎週の手動集計が負担になってきた
- 自分（Motobu）が更新を楽しいと感じ続けている

**実装内容**:
- Supabase移行（無料プラン）
- ログイン不要の匿名認証
- 端末跨ぎ同期
- リアルタイムランキング

**移行手順の目安**:
1. Supabaseでテーブル設計（30分）
2. データ読み書き処理を差し替え（数時間、コード構造を整えていれば）
3. 既存ユーザーに「JSONエクスポート→アップロード」を案内
4. 並行運用しながら問題なければLocalStorage廃止

**推定工数**: 1〜2日

## 6. 運用ルール

### 6.1 曲の追加
1. `data/songs.json` に新エントリを追加
2. ID は連番で採番（156の次は157）
3. タグ付けに迷ったら Perplexity / Gemini で雰囲気を確認（プロンプトはspecification.md内に格納予定）
4. git commit で履歴管理
5. デプロイ（GitHub Pagesは自動）

### 6.2 仕様変更
- 大きな変更時は specification.md と本書を両方更新
- 設計判断を変える時は本書に「変更理由」を追記

### 6.3 トラブル時の優先順位
1. **ゑうたん本人・事務所からの懸念** → 最優先で対応、必要なら閉鎖も辞さない
2. **リスナーからの不具合報告** → できる範囲で迅速対応
3. **機能要望** → Phase 2/3 のロードマップに照らして判断

## 8. プロジェクトの「らしさ」を保つために

このプロジェクトには明確な「らしさ」がある：

- **HSP配慮** — 派手にしない、強制しない、刺激を抑える
- **一ファンとしての姿勢** — 公式じゃない、勝手に作っている、本人を尊重する
- **長期運用を見据えた軽量設計** — 重くしない、最初から完璧を目指さない
- **「歌ってみた」文化への理解** — 同じ曲を何度でも聴きたい気持ちを尊重

これらの「らしさ」が、機能追加やデザイン変更の判断軸になる。
迷ったら本書に戻って、当初の意図を確認すること。
